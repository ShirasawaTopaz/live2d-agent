# Contributing to Live2oder

Thank you for your interest in contributing to Live2oder! We welcome contributions from everyone, whether you're fixing a bug, adding a feature, or improving documentation. This document will help you get started.

## Code of Conduct

By participating in this project, you agree to:
- Be respectful and inclusive of all people regardless of background
- Communicate professionally, even when disagreeing
- Focus on what is best for the community and the project
- Avoid personal attacks, harassment, or toxic behavior

If you experience or witness unacceptable behavior, please report it to the project maintainers.

## How to Contribute

### Reporting Bugs

If you find a bug:
1. First, check the [issues](../../issues) to see if it has already been reported
2. If not, open a new issue
3. Include a clear title and description of the problem
4. Provide steps to reproduce the issue
5. Include information about your environment (OS, Python version, etc.)
6. Add screenshots or error logs if helpful

Good bug reports make it much easier to fix issues quickly.

### Suggesting Features

We love new ideas! If you have a feature suggestion:
1. Check existing issues to see if someone else has already suggested it
2. Open a new issue with the tag "enhancement"
3. Clearly describe the feature and why it would be useful
4. Provide examples of how it would work if possible

### Submitting Pull Requests

Ready to contribute code or documentation changes? Follow these steps:

1. Fork the repository
2. Create a new branch for your work (`git checkout -b feature/my-new-feature`)
3. Make your changes following the coding guidelines below
4. Test your changes to make sure they work correctly
5. Commit your changes with a clear descriptive message
6. Push your branch to your fork
7. Open a pull request to the main repository

## Development Process

1. **Fork** the repository on GitHub
2. **Clone** your forked repository to your local machine
3. **Create** a feature branch from `main` (`git checkout -b feature/your-feature-name`)
4. **Make** your changes, following coding guidelines
5. **Test** your changes locally
6. **Commit** your changes with a descriptive commit message
7. **Push** your branch to your fork on GitHub
8. **Open** a pull request to the main repository
9. **Wait** for review and address any feedback
10. Your pull request will be merged once approved

## Coding Guidelines

- Follow the existing coding style and patterns you see in the project
- Use meaningful variable and function names
- Keep functions focused on a single responsibility
- Add docstrings for new classes and functions
- If you're adding a new feature, include tests when possible
- Update documentation if your changes add or change functionality
- Keep pull requests focused on one change - large pull requests are harder to review

Live2oder is written in Python with PySide6, following PEP 8 style guidelines where practical.

## Documentation

If you add or change any functionality:
- Update the README.md if necessary to reflect new features
- Add comments in your code where it helps understanding
- Update or add documentation for new configuration options
- If you add a new tool or model backend, document how to use it

Good documentation makes the project better for everyone.

## Review Process

After you submit a pull request:
1. A maintainer will review your code
2. We may ask for changes or clarifications - this is a normal part of the process
3. Once all feedback is addressed and the changes are approved, your pull request will be merged
4. We aim to review pull requests within a few business days

We want to make sure every contribution aligns with the project goals and maintains good code quality. Don't be discouraged if we ask for changes - this helps make the project better.

## Getting Help

If you have questions about contributing, feel free to open an issue with your question, and we'll help you get started.

Thank you again for contributing!
